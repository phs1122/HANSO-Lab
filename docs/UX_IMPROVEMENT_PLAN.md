# HANSO Lab UX 개선 계획

문서 목적: 사용자가 제기한 6개 UX 항목을 구조화해 후속 구현·논의의 기준으로 삼는다.
이 문서는 "무엇을 왜 바꾸는가"의 합의용이며, 세부 구현 지시서는 Phase별로 따로 만든다.
작성: 2026-07-09.

> ⚠ **2026-07-17 개정**: 연결 방식·케이블·레벨 정책은
> `docs/CAPTURE_CONNECTION_POLICY.md`가 정본이다. 본 문서의 해당 서술
> (단일 TS 케이블 권장, -12 dBFS 고정, 스테레오 표기 등)은 정본에 맞게
> 아래에서 수정되었으며, 충돌 시 정본을 따른다. 핵심 변경: 간편 캡쳐는
> 리앰프 박스의 "대체"가 아니라 직접 연결을 보조하는 **호환 모드**이고,
> 헤드폰 직결의 기본 케이블은 **TRS→듀얼 TS 브레이크아웃(한쪽 플러그만)**,
> 단일 TS 직결은 공식 미지원(실험적)이다. 두 모드 모두 저레벨 시작
> 단계적 캘리브레이션을 사용한다.

---

## 0. 핵심 진단

지금 앱을 열면 곧바로 "Capture Wizard" 화면이 던져진다. 사용자는 다음을 예측할 수 없다:

- 이 앱이 무엇을 하는가
- **내 장비로 무엇을 할 수 있는가**
- 간편/일반 중 무엇을 골라야 하는가 (이름만으로는 의미 불명 → 문진으로 자동 결정)
- 각 단계를 왜/어떻게 하는가

즉 근본 문제는 개별 위젯이 아니라 **온보딩(사용자 환경 파악 + 설명)의 부재**다.

---

## 1. 목표 플로우 (개선 후)

```
앱 실행
  → [환영 + 개요]      이 앱이 뭘 하나 (1~2문장)
  → [무엇을 캡쳐?]     Amp head / Combo / Cabinet / Pedal(추후)
  → [장비 문진]        보유 장비 체크 → 가능한 캡쳐 경로 판정
  → [결과 / 추천]      "당신 장비로는 ~ 방식 가능" + 부족 장비 안내 + 모드 자동 결정
  → [튜토리얼]         선택한 캡쳐의 원리 · 신호 체인 · 인터페이스 세팅
  → [캡쳐 워크플로우]   (기존 Setup → Calibration → Gain → Finish)
```

현재는 이 중 마지막 줄(캡쳐 워크플로우)만 존재하고 앞의 온보딩이 전부 빠져 있다.

## 2. 항목별 정리

### 항목 1 — "Capture Wizard" 타이틀 제거
- 앱은 이미 상단에 "HANSO Lab"으로 타이틀됨. "Capture Wizard"는 중복·기계적.
- 대체안: 현재 캡쳐 대상 + 진행 단계를 보여주는 맥락 헤더, 또는 제거 후 여백 확보.
- 난이도: **낮음. 즉시 적용 가능.**

### 항목 2 — Combo 캡쳐 타입 추가
- 현재 타입: Amp, Cabinet (+ 추후 Pedal/Effect/FullRig).
- **Combo = 앰프 + 내장 스피커 일체형.** 스피커를 마이킹해 앰프+스피커 통합음을 캡쳐.
- 타입별 신호 체인/장비 차이:

  | 타입 | 캡쳐 출력원 | 필요 장비 |
  |---|---|---|
  | Amp (head) | 라인/DI/로드박스 | 로드박스 또는 라인아웃 |
  | **Combo** | **스피커 마이킹** | **마이크** (+ 리앰프 or 헤드폰 직결 입력) |
  | Cabinet | 스피커 IR | IR 소스 + 마이크 |

- 결과물: amp+cab 결합 모델 (NAM의 `amp_cab` gear_type에 해당).
- ✅ **결정 (2026-07-09): Amp 워크플로우 재사용.** Setup→Calibration→Gain→Finish를 그대로 쓰되, **Setup 단계에 스피커 마이킹 배치 안내를 추가**하고, 결과 모델을 **amp+cab(스피커 포함)로 태그**한다(별도 캐비닛과 짝짓지 않음). 새 상태머신 없음. 입력 경로는 모드(간편/일반)대로, 출력은 마이크.

### 항목 3 & 5 — 워크플로우 설명 + 퍼-모드 튜토리얼
각 캡쳐 타입/모드 진입 시 튜토리얼 화면 제공:

1. **원리** — 이 캡쳐가 무엇을 측정/모델링하나 (테스트 신호 → 전달 특성 → 모델).
2. **필요 장비 체크리스트**.
3. **신호 체인 다이어그램** — 인터페이스 → (리앰프) → 장비 → 리턴.
4. **오디오 인터페이스 세팅** — 샘플레이트, 버퍼, 입출력 라우팅, 게인 스테이징, 믹서 루프백 뮤트.

- "다시 보지 않기" 체크 + 도움말 아이콘으로 재접근.
- 난이도: **중** (핵심은 콘텐츠 작성).

### 항목 4 — 온보딩 문진 + 모드 네이밍 재고

**문진(질문지)** — ✅ **확정 (2026-07-09).** 사용자가 모드를 "고르는" 게 아니라 장비를 "체크"하면 앱이 판정한다. 라벨은 선택지가 아니라 "당신 장비로는 이 방식"이라는 설명이다.

```
STEP 0 — 필수 게이트
  오디오 인터페이스가 있나요?
    ├ 예   → 계속
    └ 아니오 → ⛔ "캡쳐에는 오디오 인터페이스가 필요합니다" (진행 차단)

STEP 1 — 무엇을 캡쳐하나요?  (캡쳐 타입)
    앰프 헤드 / 콤보 앰프 / 캐비닛 / 페달(추후)

STEP 2 — 입력 경로 (← 간편/일반 결정)
  Q2a. 리앰프 박스가 있나요?
    ├ 예   → 일반 캡쳐 (Line Out → 리앰프 → 장비 / 모노 신호 / 저레벨 캘리브레이션)
    └ 아니오 → Q2b
  Q2b. (리앰프 없음) 인터페이스 출력은?
    ├ 라인 아웃 있음 → 간편 캡쳐 · 라인 직결
    │                   (단자 형태(TS/TRS/XLR/RCA)에 따라 배선 확인 필요
    │                    · 험/그라운드 루프 주의)
    └ 헤드폰만       → 간편 캡쳐 · 헤드폰 직결 (Left-only, 슬라이더로 드라이브)
        └ 케이블 확인 질문 (2026-07-18 확정 · Progressive Disclosure):
            ✅ 기본: TRS→듀얼 TS 브레이크아웃 — LEFT(L) 플러그만 앰프에,
                     RIGHT(R) 플러그는 미연결 · 3.5mm면 3.5mm→6.35mm 어댑터
            ⚠ "일반 기타 TS 케이블만 있음" 선택 시에만
              → Experimental Quick Start: 위험 고지 인터스티셜 + 명시적
                동의를 거쳐 진행 (한 클릭 선택 금지, 정본 §2)

STEP 3 — 출력 캡쳐 경로 (캡쳐 타입별)
  [앰프 헤드] 앰프 소리를 어떻게 받나요?
    로드박스 / 앰프의 라인·DI·이펙트 루프 센드 / 없음
    ⚠ 스피커 아웃을 인터페이스에 직결 금지
  [콤보 / 캐비닛] 마이크가 있나요?  (캐비닛은 + 파워앰프/신호원)
    있음 / 없음

STEP 4 — 판정
    ✅ 다 갖춤 → 모드 확정 → 튜토리얼로
    ⚠ 부족   → 무엇이 왜 필요한지 + (가능하면) 임시 대안
```

**판정 로직 요약**
```
인터페이스 없음                    → 캡쳐 불가
리앰프 박스 있음                   → 일반 캡쳐
리앰프 없음 + 라인 아웃            → 간편 캡쳐 · 라인 직결 (배선 확인)
리앰프 없음 + 헤드폰만             → 간편 캡쳐 · 헤드폰 직결
                                     (브레이크아웃 LEFT만 · TS만 있으면
                                      Experimental Quick Start 동의 경유)
[앰프헤드] 로드박스/라인아웃 없음  → "출력 캡쳐 장비 필요"
[콤보/캐비닛] 마이크 없음          → "마이크 필요"
```

**설계 원칙**
- 모드는 "리앰프 박스 유무" 한 축으로만 갈린다 (간편/일반). 라인/헤드폰 차이는 간편 안의 **하위 경로**이며, 앱이 라우팅·경고로 처리한다 (3번째 모드 만들지 않음).
- 케이블 안내는 **헤드폰 직결 분기**에서 표시. 기본은 `TrsToDualTsYCable`
  (브레이크아웃, 한쪽 플러그만), `NormalTsCable`(단일 TS 직결)은 공식 미지원·
  실험적으로 강등 (2026-07-17 개정). 라인 직결도 단자 형태에 따라 배선 확인이
  필요하므로 "TS-TS 단순"으로 일반화하지 않는다 → 정본 §5.
- 간편 캡쳐는 리앰프 박스의 대체가 아니라 직접 연결 **호환 모드**로 서술한다.
- 막다른 길에도 이유와 대안을 준다.
- 문진 확장(출력 단자·케이블 종류 + 단자 그림, 위험 연결 차단)은 후속 Phase
  → 정본 §4, §8.

**모드 네이밍** — ✅ **결정됨 (2026-07-09): 간편 캡쳐 / 일반 캡쳐.**
("정식"은 품질 등급처럼 들려 오해를 부르므로 "일반"으로 변경. 내부 enum은 `Easy`/`Standard` 유지, 한국어 표시만 변경.)

- 간편 캡쳐(Easy) = 리앰프 박스 없음, 라인/헤드폰 직결(호환 모드), 모노,
  출력 슬라이더로 조절 (저레벨 시작).
- 일반 캡쳐(Standard) = 리앰프 박스, 라인아웃, 모노 신호(듀얼 모노 출력),
  저레벨 시작 단계적 캘리브레이션 (-12 dBFS는 고정값이 아니라 초기 제안값
  — 2026-07-17 개정, 정본 §3).

문진 결과로 모드가 결정되므로, 라벨은 "선택지"가 아니라 "설명"으로 쓴다.

### 항목 6 — UI 폴리시 & 접근성
- 정보 위계: 제목 / 안내 / 경고 / 액션 버튼의 시각적 구분.
- 진행 표시: 현재 단계가 전체 중 어디인지 (스텝 인디케이터).
- 여백·타이포·색 대비 정리 (경고=빨강, 정상=초록의 의미 일관).
- 안내문 길이/톤 정리 (긴 단락 → 요점 + 접기).
- **보존 주의**: `docs/CURSOR_FIXES.md`의 calibration 우측 컬럼/미터 레이아웃 픽스를 깨지 말 것.

### 온보딩 사양 (결정 4·5 종합)
- **형태**: 전체화면 별도 시퀀스 (환영 → 문진 → 결과/추천 → 튜토리얼) → 완료 후 기존 캡쳐 화면으로 진입. 캡쳐 화면은 깔끔하게 유지.
- **노출**:
  - 첫 실행: 전체 온보딩.
  - 장비 문진 답변은 **저장**해 재사용.
  - 이후 새 캡쳐: "무엇을 캡쳐?"만 빠르게. 장비가 바뀌면 "환경 다시 설정"으로 문진 재실행.
  - 온보딩/튜토리얼은 **재접근 버튼** 상시 제공.
- 구현: 신규 `OnboardingPanel`(가칭)이 시퀀스를 돌리고 완료 시 `CapturePanel`로 핸드오프. 저장은 `ApplicationState`(또는 설정 파일).

## 4. 제안 구현 순서

| Phase | 내용 | 비용/효과 |
|---|---|---|
| 1 | "Capture Wizard" 제거 + 캡쳐별 워크플로우 설명 텍스트 | 저비용·즉효 |
| 2 | 장비 문진 + 모드 자동결정 + 네이밍 반영 | 중 |
| 3 | 퍼-모드 튜토리얼 (원리·체인·인터페이스 세팅) | 중 (콘텐츠) |
| 4 | Combo 캡쳐 타입 + 워크플로우 | 중~고 |
| 5 | UI 폴리시 (위계·타이포·여백·진행표시) | 중 |

각 Phase는 독립 논의·구현 가능. Phase 1은 즉시 착수 가능.
