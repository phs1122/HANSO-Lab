# HANSO Lab 캡쳐 연결·레벨 정책 (정본)

작성: 2026-07-17. 개정: 2026-07-18 (검토 반박 6건 결정 반영 — LEFT-only 고정,
메타데이터 스키마 교체, TS 근거 축소, Experimental Quick Start 계층,
사용자 승인 캘리브레이션, Progressive Disclosure).
지위: **이 문서가 캡쳐 연결 방식·레벨 정책·관련 UX 문구의 정본이다.**
`docs/UX_IMPROVEMENT_PLAN.md`(2026-07-09)의 문진·케이블·레벨 서술 중 이 문서와
충돌하는 내용은 이 문서가 대체한다.

**제품 철학 요약**: Standard Capture는 더 재현성 높은 권장 경로이고, Easy
Capture는 장비 진입 장벽을 낮추는 경로다. Easy를 안전하게 만들기 위해 모든
비표준 연결을 없애는 것이 아니라, Recommended와 Experimental을 UX 계층으로
명확히 구분해 사용자가 가장 안전한 경로를 먼저 선택하도록 유도하되, 장비가
없는 사용자의 진입 자체를 막지는 않는다. — "사용자 경험은 쉽고 단순하게,
결과물과 내부 데이터는 전문적으로."

---

## 0. 설계 전제 — 불특정 다수의 하드웨어

HANSO Lab은 개인용 도구가 아니라 배포를 전제로 하는 애플리케이션이다.
설계 기준은 개발자의 보유 장비나 특정 사용자의 캡쳐 환경이 아니라, **불특정
다수가 사용하는 다양한 오디오 인터페이스·케이블·드라이버·라우팅 환경**이다.

> "내 환경에서 동작한다"는 사실은 제품 차원의 안전성, 호환성, UX를 보증하지
> 못한다.

여기서 나오는 원칙:

- 특정 인터페이스(예: 개발자의 Babyface Pro)나 특정 리앰프 박스를 기준으로
  로직을 고정하지 않는다.
- 오디오 인터페이스별 최대 출력 전압과 출력 회로 구조를 **앱이 알 수 없다는
  전제**에서 설계한다.
- 배포 제품의 UX는 특정 장비에서 작동하는 최단 연결이 아니라, **다양한
  사용자가 잘못 연결하더라도 사고 가능성을 최소화하는 방향**으로 설계한다.

## 1. 간편 캡쳐의 정의 — 대체 모드가 아니라 호환 모드

소프트웨어가 할 수 있는 것:

- 디지털 출력 레벨 조절
- 좌우 채널 라우팅
- 테스트 신호의 진폭 제한
- 캘리브레이션 절차 제공
- 반환 신호의 클리핑 감지

리앰프 박스가 하드웨어로 수행하는 것 (소프트웨어로 대체 불가):

- 밸런스드 라인 신호 → 언밸런스드 신호 변환
- 소스 임피던스를 앰프/페달 입력에 적합하게 조정
- 라인 레벨을 인스트루먼트 입력에 적합한 범위로 감쇠
- 트랜스 절연 등으로 그라운드 루프 저감
- 인터페이스 출력단과 앰프 입력단 사이의 전기적 조건 표준화

**금지 표현**: "리앰프 박스의 역할을 앱이 대신한다."

**올바른 표현**: "앱은 리앰프 박스 없이 직접 연결하는 환경에서 출력 레벨 관리,
라우팅, 캘리브레이션을 보조한다. 임피던스 변환, 밸런스 변환, 절연과 같은
리앰프 박스의 물리적 기능은 대신하지 않는다."

간편 캡쳐 = **비표준 직접 연결을 가능한 범위에서 보조하는 호환 모드.**
리앰프 박스의 대체 모드가 아니다.

## 2. 연결 계층 — Recommended / Standard / Experimental Quick Start

### Standard (권장 · 리앰프)

```
Interface Line Out → Reamp Box → Amp Input
```

### Recommended (간편 캡쳐의 기본 경로)

언밸런스드 라인 출력 직접 연결:

```
Interface Unbalanced Line Out → 적합한 TS 케이블 → Amp Input
```

헤드폰 출력 직접 연결 — **LEFT 플러그만 사용** (2026-07-18 확정):

```
Interface Stereo Phones Out
→ TRS-to-Dual-TS 브레이크아웃 케이블
→ LEFT TS 플러그만 사용
→ Amp Input
```

앱의 헤드폰 경로 라우팅은 Left-only로 고정한다 (`L = Capture Signal,
R = Digital Mute`). UI 안내 문구: **"브레이크아웃 케이블의 LEFT(L) 플러그만
앰프 INPUT에 연결하세요. RIGHT(R) 플러그는 연결하지 마세요."** 양 채널에
동일한 모노 신호를 내보내는 방식은 채택하지 않는다 — 사용하지 않는 R
플러그에도 신호가 실려 불필요한 활성 출력이 하나 더 생기고, 연결 실수
가능성만 증가한다.

### Experimental Quick Start (공식 기본 연결 아님)

```
Interface Stereo Phones Out → 단일 TS 케이블 직접 삽입 → Amp Input
```

기본 UX에서 제외하는 핵심 근거는 **다음 두 가지로 한정한다** (2026-07-18에
근거 목록 축소 — 삽입/제거 순간의 접점 전환은 TRS 케이블에도 공통이므로
고유 근거가 아님):

- **A. 정상 연결 상태에서 R 출력이 지속적으로 Ground에 연결될 수 있음**
  (Tip=Left, Ring=Right, Sleeve=Ground에 모노 TS 삽입 시 Ring→Sleeve)
- **B. HANSO Lab이 해당 R 출력의 모든 상태를 통제할 수 없음**
  (다른 애플리케이션, OS 출력, 인터페이스 DSP 믹서, 앱 종료 후 상태,
  드라이버 재초기화)

위험 서술 규정: "매우 위험하여 장비가 손상될 수 있다"가 아니라 —
**"대부분의 현대적인 헤드폰 출력에서 즉각적인 손상 가능성은 높지 않을 수
있으나, HANSO Lab이 불특정 인터페이스의 출력 회로와 보호 설계를 보증할 수
없으므로 공식 기본 연결로 제공하지 않는다."** 이것은 위험 과장이 아니라
제품 지원 범위의 문제다.

**완전 삭제하지 않는 이유**: "장비가 부족해도 일단 캡처를 경험할 수 있다"는
간편 캡쳐의 핵심 UX 가치를 지키기 위해서다. 다만 진입은 반드시 **별도
인터스티셜(위험 고지) + 명시적 동의**를 거친다:

```
실험적 연결 안내 (고지문)
  [권장 연결 방법 보기]
  [위 내용을 이해했으며 실험적 연결로 계속]
```

단순 드롭다운 한 클릭으로 선택되는 구현은 금지한다. 온보딩에서는 "일반 기타
TS 케이블만 있음"을 선택한 경우에만 이 경로가 나타난다.

## 3. 레벨 정책 — 고정 dBFS 금지, 단계적 레벨 탐색

dBFS는 디지털 신호의 크기일 뿐이다. 실제 아날로그 출력 전압은 인터페이스
모델, Line/Phones 여부, +4 dBu·−10 dBV 기준, 최대 출력 전압, 하드웨어 볼륨
노브, DSP 믹서 감쇠, 드라이버/OS 볼륨, 밸런스드·언밸런스드 연결, 케이블
배선에 따라 달라진다. **같은 -12 dBFS라도 앰프 입력에 도달하는 전압은 장비마다
크게 다르다.**

또한 패시브 리앰프 박스는 증폭하지 못하고 출력 노브도 감쇠 범위에서
동작하므로, "출력은 -12 dBFS 고정, 드라이브는 리앰프 노브로"라는 정책은 출력이
부족한 인터페이스에서 성립하지 않는다.

**일반 캡쳐와 간편 캡쳐 모두** 저레벨 시작 단계적 캘리브레이션을 사용한다.
단, **자동 출력 상승은 금지한다** (2026-07-18 확정). 100W 진공관 앰프 +
4×12 캐비닛 같은 환경에서 소프트웨어가 스스로 출력을 올리는 것은 그 자체가
위험이다. 절차는 항상 사용자 승인 기반이다:

```
앱: 낮은 테스트 레벨 제안
→ 사용자: 출력 증가 허용
→ 앱: 짧은 테스트 신호 출력
→ 앱: Return Level 분석
→ 사용자: 다음 단계 증가 승인
→ (반복) → 사용 가능한 캡쳐 레벨 확정
```

즉 **Software-assisted calibration**이지 Automatic gain sweep이 아니다.

`-12 dBFS`는 절대 기준이 아니며 다음으로만 사용한다:

- 초기 제안값
- 장치 프로파일이 확인된 경우의 기준값
- 캘리브레이션 과정에서 변경 가능한 상한 후보

현재 구현: 캘리브레이션은 무음(노이즈 플로어 측정) → 저레벨 프로브 순으로
진행하고, 반환 창(-36 ~ -8 dBFS)·SNR·톤 우세도 검사를 통과해야 한다. 출력
슬라이더는 두 모드 모두 노출되며 낮은 기본값(-33 dBFS)에서 시작한다 —
수동 슬라이더 + 반환 창 검증 구조가 이 원칙에 부합하므로 유지한다. 향후
UX 개선은 슬라이더 제거가 아니라 `[테스트] → 결과 표시 → [조금 높이기]`
형태의 사용자 승인 단계식 인터페이스 방향으로 한다.

## 4. 장비 문진 — "리앰프 박스가 있나요?" 하나로 끝내지 않는다

배포 환경에서는 최소한 다음을 구분해야 한다: 밸런스드 XLR/TRS Line Out,
언밸런스드 TS Line Out, RCA Line Out, 6.35mm/3.5mm 스테레오 Phones Out,
인터페이스 DSP 믹서 사용 여부, 출력 볼륨의 소프트웨어/하드웨어 제어 여부,
리앰프 박스의 패시브/액티브 및 입력 단자, 사용자가 가진 케이블 종류.

목표 문진 순서:

```
1. 리앰프 박스 보유 여부
2. 사용할 인터페이스 출력 단자 선택
3. 해당 출력의 단자 형태 확인
4. 사용할 케이블 선택
5. 앱이 연결 가능 여부 판정
6. 지원 연결만 단계별 안내
7. 위험하거나 비표준적인 연결은 차단 또는 경고
```

**Progressive Disclosure 원칙 (2026-07-18 확정)**: 질문 개수를 줄이는 것이
아니라, 질문을 한꺼번에 보여주지 않는다. `Question Tree ≠ Long
Questionnaire`. 사용자는 자신의 선택에 필요한 질문만 순차적으로 보고,
내부적으로는 필요한 정보를 모두 수집한다. 예: 헤드폰 선택 시에만 케이블
질문이 나타나고, "일반 TS 케이블만 있음"을 선택한 경우에만 Experimental
Quick Start 경로가 나타난다. 설계 기준은 여전히 **"사용자는 전기공학 지식
없이 연결할 수 있어야 한다"**이다.

사용자가 기술 용어를 모를 수 있으므로 단자 이름만 묻지 말고 **단자 그림/사진을
함께 제공**한다. (그림 제공은 구현 Phase — §8 참고. 케이블 확인 질문과
Experimental 동의 화면은 2026-07-18 구현 완료.)

## 5. 라인 아웃 직결의 배선 다양성

밸런스드 출력을 언밸런스드 입력에 연결하는 권장 배선은 제조사·회로에 따라
다르다 (Cold를 Ground에 연결 / 미연결 / 저항 경유 / 전용 케이블). 따라서
`Line Out → 일반 TS 케이블 → Amp Input`으로 일반화해 안내하지 않는다.

- RCA 또는 명시적 언밸런스드 TS 출력 → 직접 연결 경로로 안내 가능
- 밸런스드 TRS/XLR 출력 → 케이블 배선 확인 필요
- 출력 구조를 알 수 없으면 보수적인 연결만 안내
- 불명확한 경우 리앰프 박스 또는 절연·언밸런스 변환 장비 사용 권장
- 케이블 이름뿐 아니라 실제 결선 조건을 내부 데이터로 관리

장기적으로 **오디오 인터페이스별 출력 프로파일 데이터베이스**가 필요하다.

## 6. 측정 경계 — 모든 캡쳐는 물리적 체인 전체를 측정한다

어떤 모드든 측정 대상은 순수한 장비(DUT)가 아니라 물리적 캡쳐 체인 전체다:

```
Standard: Interface Output → Reamp Box → DUT → Return Chain → Interface Input
Easy:     Interface Output → Direct Connection → DUT → Return Chain → Interface Input
```

일반 캡쳐도 리앰프 박스(트랜스 특성, 레벨 노브)와 반환 체인(마이크, 프리앰프
등)을 함께 측정하므로, Standard를 `"measurementBoundary": "device"`로 기록하는
것은 지나치게 이상화된 표현이다 (2026-07-18 폐기). **리앰프 박스는 절대적인
기준 신호가 아니라, 직결보다 재현성과 전기적 조건의 표준화 가능성이 높은
방식으로 취급한다.** 따라서 제품 설명도 "리앰프 캡처는 표준화된 캡처"가
아니라 — "리앰프 캡처는 직접 연결 방식에 비해 입력 조건의 재현성과 장비 간
호환성을 높일 수 있는 권장 방식입니다."

메타데이터는 **Target(무엇을 모델링하려 했나)과 Boundary(데이터가 실제로
어떤 물리 체인을 통과했나)를 분리**해 기록한다:

```json
{
  "captureTarget": "Amp",
  "measurementBoundary": "physical_capture_chain",
  "excitationPath": { "mode": "reamp | direct_line | direct_headphone" },
  "returnPath":     { "mode": "line_or_di | loadbox_line | microphone | instrument_line" }
}
```

모드 간 실제 차이는 `excitationPath.mode`와 `returnPath.mode`가 표현한다.
향후 리앰프 제품별 특성도 확장 기록할 수 있다:

```json
"excitationPath": {
  "mode": "reamp",
  "manufacturer": "Radial",
  "model": "JCR",
  "userSpecified": true
}
```

같은 앰프라도 다른 인터페이스로 직결 캡쳐하면 결과가 달라질 수 있다. 빈티지
앰프, 퍼즈 페달, 게르마늄 회로, 패시브 픽업과의 상호작용을 전제한 페달, 입력
임피던스가 낮거나 비선형적인 회로에서 특히 민감하다.

**부족한 문구**: "리앰프 방식보다 정확도가 낮을 수 있습니다."

**규정 문구**: "간편 캡쳐는 인터페이스와 직접 연결 조건의 영향을 함께 포함할
수 있으므로, 리앰프 기반 캡쳐보다 장비 간 재현성과 입력 조건의 표준화 수준이
낮습니다."

이 차이는 단순 품질 열화가 아니라 **모델링 대상의 경계가 달라지는 문제**다.

## 7. 기타 문구·설계 규정

### 모노 기본

일반적인 기타 앰프 입력과 리앰프 박스 출력은 모노다. 기본 캡쳐 경로는:

```
모노 테스트 신호 → 모노 앰프 입력 → 모노 또는 다중 마이크 반환
```

- UI에서 "일반 캡쳐: 스테레오", "라인 아웃 직결: 스테레오 가능" 같은 표현
  금지. (Standard의 Default 라우팅은 같은 모노 신호를 양 채널에 내보내는
  듀얼 모노 편의이지 스테레오 캡쳐가 아니다.)
- 마이크 두 개 사용 = 다중 입력 캡쳐 (스테레오 출력 캡쳐가 아님).
- 앰프 두 대 동시 구동 = 별도의 멀티앰프 모드로 분리 (미구현).

### 헤드폰 출력 서술

헤드폰 출력은 인터페이스에 따라 최대 전압·전류·출력 임피던스·보호 회로·볼륨
범위·단자별 출력이 크게 다르며, 라인 출력만큼 높거나 더 강할 수 있다.

- **금지**: "헤드폰 직결은 출력이 약합니다."
- **규정**: "헤드폰 출력은 장치별 최대 레벨과 출력 회로의 차이가 크기 때문에,
  반드시 낮은 출력에서 캘리브레이션을 시작해야 합니다."

### 제품 차원의 핵심 주장 (이 수준으로 제한)

> HANSO Lab은 리앰프 박스가 없는 사용자도 일부 오디오 인터페이스와 앰프
> 조합에서 캡쳐를 시도할 수 있도록 간편 캡쳐 모드를 제공한다. 앱은 출력
> 라우팅, 저레벨 캘리브레이션, 진폭 제한, 클리핑 검사와 연결 안내를
> 자동화한다. 다만 소프트웨어는 리앰프 박스의 임피던스 변환, 밸런스 변환 및
> 전기적 절연을 대체하지 못하므로, 직접 연결 방식은 일반 캡쳐보다 노이즈,
> 호환성, 재현성과 입력 표준화 수준이 낮을 수 있다.
